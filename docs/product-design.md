# talkdesk プロダクト設計

## 1. コンセプト

中小企業が庶務・経理・労務などの単純作業をオンラインアシスタントにアウトソースするための、業務単位のチャットコミュニケーションツール。

- 依頼・報告・成果物の受け渡しをすべて「業務ごとのスレッド」に集約し、過去のやり取りを辿りやすくする
- メールや汎用チャット（Slack等）と違い、**権限管理を前提にした設計**で給与情報などの機密業務も安心して依頼できる

## 2. 登場人物（ロール）

| ロール | 説明 |
|---|---|
| 依頼者（クライアントユーザー） | 顧客企業に所属。自社のスレッドのうち、閲覧権限のあるものだけ見える |
| クライアント管理者 | 依頼者のうち、自社メンバーの招待・スレッド作成・スレッド閲覧権限の管理ができる人 |
| アシスタント | 運営側に所属する人間のオペレーター。アサインされた企業のスレッドのみ見える |
| 運営管理者 | サービス運営者。企業アカウントの発行、アシスタントの企業へのアサインを行う |

※ 運営管理者は要件に明示されていないが、「アシスタントを企業にアサインする」操作を誰かが行う必要があるため設けた。

## 3. ドメインモデル

```
Organization（顧客企業）
 ├─ Membership（依頼者の所属。role: admin / member）
 ├─ Assignment（アシスタントのアサイン）
 └─ Thread（業務単位のスレッド。例:「給与計算」「経費精算」「請求書発行」）
     ├─ ThreadMember（依頼者側の閲覧権限。スレッドごとに明示的に付与）
     ├─ Message（発言。type: 依頼 / 報告 / コメント）
     │   └─ Attachment（添付ファイル・成果物）
     └─ Request（依頼カード。status: 依頼中 / 対応中 / 確認待ち / 完了）

User（依頼者・アシスタント・運営管理者を共通のUserとして持ち、roleで区別）
```

### スレッド＝業務、依頼＝カード

- スレッドは「業務」に対応する長寿命のチャンネル（例:「毎月の給与計算」）
- 個々の依頼は、スレッド内の**依頼カード（Request）**として起票し、ステータスを持たせる
- ステータス遷移: アシスタントが「依頼中→対応中→確認待ち」まで進め、依頼者が成果物を確認して「完了」を承認する
- チャットメッセージは依頼カードに紐づけて（またはスレッド直下に）投稿できる
- これにより「過去の依頼内容・報告を辿る」が、チャットログの掘り返しではなくカード一覧で実現できる

### 成果物

- アシスタントはメッセージにファイルを添付する形で成果物を納品
- スレッドごとに「ファイル」タブを設け、そのスレッドに投稿された添付を時系列で一覧できる

## 4. 権限モデル（本プロダクトの肝）

### 原則

1. **企業間の分離が最優先**: 依頼者は自社の Organization 配下のデータ以外に一切アクセスできない
2. **スレッドはデフォルト非公開**: 依頼者はThreadMemberに追加されたスレッドだけ見える（給与スレッドは経営者と労務担当だけ、など）
3. **アシスタントは企業単位**: アサインされた企業の全スレッドが見える（アシスタントは運営側の人間であり、業務遂行に必要なため）

### 権限マトリクス

| 操作 | 依頼者(member) | クライアント管理者 | アシスタント | 運営管理者 |
|---|---|---|---|---|
| 自社スレッド閲覧 | 権限のあるものだけ | 全て※ | アサイン企業の全て | 全て |
| スレッド作成 | ✕ | ○ | ○ | ○ |
| スレッド閲覧権限の付与/剥奪 | ✕ | ○ | ✕ | ○ |
| 依頼の起票・メッセージ投稿 | 閲覧可能なスレッドで○ | ○ | ○ | ○ |
| ファイル添付 | ○ | ○ | ○ | ○ |
| 自社メンバー招待 | ✕ | ○ | ✕ | ○ |
| アシスタントのアサイン | ✕ | ✕ | ✕ | ○ |

※ クライアント管理者は経営者が務める想定のため、管理者は自社の全スレッドを閲覧できる（「管理者にも見えないスレッド」は設けない）。

### 実装上の注意

- 権限チェックはUIの出し分けではなく、**API層で必ず enforce** する（スレッド一覧・メッセージ取得・ファイルダウンロードすべて）
- 添付ファイルのURLは署名付き・期限付きにし、URL共有だけでは権限外の人が開けないようにする

## 5. 主要画面

1. **ホーム（スレッド一覧）**
   - 依頼者: 自社の閲覧可能スレッド一覧＋未読バッジ
   - アシスタント: アサイン企業ごとにグルーピングされたスレッド一覧（横断ビュー）
2. **スレッド画面**
   - 上部: 業務名・参加メンバー
   - タブ: チャット / 依頼一覧（カード＋ステータス） / ファイル
3. **依頼カード詳細**: 依頼内容、期日、ステータス、紐づくやり取りと成果物
4. **メンバー管理（クライアント管理者用）**: メンバー招待、スレッドごとの閲覧権限のマトリクス編集
5. **運営管理画面**: 企業アカウント発行、アシスタントのアサイン管理

## 6. MVPスコープ

**入れる**
- 企業・ユーザー・ロール・スレッド閲覧権限（上記マトリクス）
- スレッド（チャット＋ファイル添付＋依頼カードとステータス）
- 未読管理と通知（メール＋WebPush）
- 運営管理画面（最低限: 企業発行・アサイン）

**入れない（将来）**
- 依頼テンプレート（毎月の定型依頼の自動起票）
- 稼働時間の記録・レポート、課金/契約管理
- Slack・メール連携での依頼受付
- 全文検索、監査ログのエクスポート

## 7. 技術方針（マルチプラットフォーム前提）

- フロントエンドは **React のSPA** として実装し、Web・デスクトップ・モバイルで共通利用する
  - デスクトップアプリ: Webview（Electron / Tauri 等）でSPAをそのまま搭載
  - ネイティブアプリ: Webviewベース（Capacitor 等）でSPAを搭載
  - このため、SPA側はプラットフォーム固有APIに直接依存せず、通知などは抽象化レイヤーを挟む
- 通知は **メール＋WebPush** の2チャネル。Webview展開時はWebPushをプラットフォームのプッシュ通知に差し替えられる構成にする
