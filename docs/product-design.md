# talkdesk プロダクト設計

## 1. コンセプト

中小企業が庶務・経理・労務などの単純作業をオンラインアシスタントにアウトソースするための、業務単位のチャットコミュニケーションツール。

- 依頼・報告・成果物の受け渡しをすべて「業務ごとのチャンネル」に集約し、過去のやり取りを辿りやすくする
- メールや汎用チャット（Slack等）と違い、**権限管理を前提にした設計**で給与情報などの機密業務も安心して依頼できる

## 2. 登場人物（ロール）

| ロール | 説明 |
|---|---|
| 依頼者（クライアントユーザー） | 顧客企業に所属。自社のチャンネルのうち、閲覧権限のあるものだけ見える |
| クライアント管理者 | 依頼者のうち、自社メンバーの招待・チャンネル作成・チャンネル閲覧権限の管理ができる人（経営者想定） |
| アシスタント | 運営側に所属する人間のオペレーター。アサインされた企業のチャンネルのみ見える |
| 運営管理者 | サービス運営者。企業アカウントの発行、アシスタントの企業へのアサインを行う |

## 3. ドメインモデル（チャンネル > スレッド > メッセージ の3層）

```
Organization（顧客企業）
 ├─ Membership（依頼者の所属。role: admin / member）
 ├─ Assignment（アシスタントのアサイン）
 └─ Channel（業務単位。例:「給与計算」「経費精算」。★権限管理の単位）
     ├─ ChannelMember（依頼者側の閲覧権限。チャンネルごとに明示的に付与）
     └─ Thread（話題単位。type: 依頼 / トピック）
         └─ Message（発言）
             └─ Attachment（添付ファイル・成果物）

User（依頼者・アシスタント・運営管理者を共通のUserとして持ち、roleで区別）
```

### チャンネル＝業務、スレッド＝依頼

- チャンネルは「業務」に対応する長寿命の入れ物（例:「毎月の給与計算」）。**閲覧権限はチャンネル単位で管理**
- すべてのメッセージはスレッドに属する（チャンネル直下の発言はない）
- スレッドには2種類ある
  - **依頼スレッド**: 個々の依頼。タイトル・依頼内容・期日・ステータスを持つ（旧「依頼カード」を統合）
  - **トピックスレッド**: 依頼に紐づかない質問・連絡
- ステータス遷移: アシスタントが「依頼中→対応中→確認待ち」まで進め、依頼者が成果物を確認して「完了」を承認する（差し戻しは→対応中）
- 「過去の依頼・報告を辿る」は、チャンネル内のスレッド一覧（ステータス・最終更新付き）で実現する

### 成果物

- アシスタントは依頼スレッド内のメッセージにファイルを添付する形で納品
- チャンネルごとに「ファイル」タブを設け、配下の全スレッドの添付を時系列で一覧できる

## 4. 権限モデル（本プロダクトの肝）

### 原則

1. **企業間の分離が最優先**: 依頼者は自社の Organization 配下のデータ以外に一切アクセスできない
2. **チャンネルはデフォルト非公開**: 依頼者はChannelMemberに追加されたチャンネルだけ見える（給与チャンネルは経営者と労務担当だけ、など）。スレッド・メッセージの閲覧権限はチャンネルから継承する（スレッド単位の権限は設けない）
3. **アシスタントは企業単位**: アサインされた企業の全チャンネルが見える（アシスタントは運営側の人間であり、業務遂行に必要なため）

### 権限マトリクス

| 操作 | 依頼者(member) | クライアント管理者 | アシスタント | 運営管理者 |
|---|---|---|---|---|
| 自社チャンネル閲覧 | 権限のあるものだけ | 全て※ | アサイン企業の全て | 全て |
| チャンネル作成 | ✕ | ○ | ○ | ○ |
| チャンネル閲覧権限の付与/剥奪 | ✕ | ○ | ✕ | ○ |
| スレッド作成・メッセージ投稿 | 閲覧可能なチャンネルで○ | ○ | ○ | ○ |
| ファイル添付 | ○ | ○ | ○ | ○ |
| 自社メンバー招待 | ✕ | ○ | ✕ | ○ |
| アシスタントのアサイン | ✕ | ✕ | ✕ | ○ |

※ クライアント管理者は経営者が務める想定のため、管理者は自社の全チャンネルを閲覧できる（「管理者にも見えないチャンネル」は設けない）。

### 実装上の注意

- 権限チェックはUIの出し分けではなく、**API層で必ず enforce** する（チャンネル一覧・スレッド・メッセージ取得・ファイルダウンロードすべて）
- 添付ファイルのURLは署名付き・期限付きにし、URL共有だけでは権限外の人が開けないようにする

## 5. 主要画面

1. **ホーム（3カラム）**
   - 依頼者: 左＝自社の閲覧可能チャンネル一覧＋未読バッジ、中＝選択チャンネルのスレッド一覧、右＝選択スレッドのメッセージ
   - アシスタント: **SlackのWorkspace切り替えUI**を踏襲。画面左端に担当企業のアイコンを縦に並べたレールを置き、企業を選ぶとその企業のチャンネル一覧に全体が切り替わる。企業アイコンには未読バッジを表示（依頼者は1社所属なのでレールは表示しない）
2. **スレッド画面**: 依頼スレッドはタイトル・ステータス・期日をヘッダーに表示し、メッセージと成果物が続く
3. **チャンネルのファイルタブ**: 配下スレッドの全添付を時系列一覧
4. **メンバー管理（クライアント管理者用）**: メンバー招待、チャンネルごとの閲覧権限のマトリクス編集
5. **運営管理画面**: 企業アカウント発行、アシスタントのアサイン管理

## 6. MVPスコープ

**入れる**
- 企業・ユーザー・ロール・チャンネル閲覧権限（上記マトリクス）
- チャンネル > スレッド > メッセージ（依頼スレッドのステータス管理、ファイル添付）
- 未読管理と通知（メール＋WebPush）
- 運営管理画面（最低限: 企業発行・アサイン）

**入れない（将来）**
- 依頼テンプレート（毎月の定型依頼の自動起票）
- 稼働時間の記録・レポート、課金/契約管理
- Slack・メール連携での依頼受付
- 全文検索、監査ログのエクスポート

## 7. 技術方針（マルチプラットフォーム前提）

- フロントエンドは **React のSPA（CSR）** として実装し、Web・デスクトップ・モバイルで共通利用する
  - SSRは採用しない（全画面ログイン必須でSEO不要、Webview同梱にはCSRが有利、チャットはクライアント側状態が主役のため）。サービス紹介LPが必要になったら別途静的サイトとして作る
  - デスクトップアプリ: Webview（Electron / Tauri 等）でSPAをそのまま搭載
  - ネイティブアプリ: Webviewベース（Capacitor 等）でSPAを搭載
  - このため、SPA側はプラットフォーム固有APIに直接依存せず、通知などは抽象化レイヤーを挟む
- 通知は **メール＋WebPush** の2チャネル。Webview展開時はWebPushをプラットフォームのプッシュ通知に差し替えられる構成にする

## 9. 決定済みの前提

- アシスタント1人あたりの担当は**最大10社**。Slack風のワークスペースレール（左端の企業アイコン列）は10社まで快適に収まる
- フロントエンドは**CSR**（SSR不採用）。構成は Vite + React のSPA ＋ APIサーバー（REST/WebSocket）の完全分離
- 情報構造は**チャンネル > スレッド > メッセージの3層**。権限管理はチャンネル単位、依頼はステータス付きスレッドとして表現
